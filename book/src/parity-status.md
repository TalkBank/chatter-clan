# Parity Status

**Status:** Current
**Last updated:** 2026-06-04 21:41 EDT

## Summary

**`chatter clan` is in active development. No single CLAN command yet
achieves byte-for-byte CLAN parity**, not FREQ, not MLU, not CHECK,
not anything else. The current snapshot below is honest about that.

If you are migrating from legacy CLAN and need bit-exact output today,
you must keep running legacy CLAN. If you are evaluating `chatter
clan` for future use, this page is the truthful description of how
far along we are and which command is under active depth-first work.

The strategy we follow is **depth-first, one command at a time**: pick
ONE CLAN command, drive it to byte-for-byte parity across every flag,
every output mode, every error path, and only then move to the next.
A flag fix scattered across many commands looks like progress without
producing any command an actual CLAN user can substitute for the
legacy tool. See [the strategy
policy](https://github.com/TalkBank/chatter/blob/main/CLAUDE.md) for
the full rationale and decision rules.

## Active command, FREQ, DEPTH-COMPLETE (2026-06-04)

Per the depth-first strategy, FREQ was the first command driven to full
parity, and it is now **depth-complete**: every FREQ flag-row is in one
of Done (byte-parity or rejects-matching-CLAN), Partial (a chatter
abstraction differs, with the residual sub-flags themselves
deferred-with-reason), Rewriter-only (errors at parse, correct for an
inapplicable flag), or Deferred (scope-gated with a source-grounded
reason). **Zero rows remain "Missing" without a source-grounded reason.**

The deferred rows are all genuine scope gates, not effort gates: `%mor`
/ cross-tier flags that need the shared MOR-analysis subsystem
(`+d0`/`+d6`/`+d7`/`+d8`, bare `+d`, `+r8`, `+r50`, `+c5`); flags that
switch CLAN to a non-CHAT raw-text mode chatter has no analog for (`+y`,
`+o2`); a CA repeat-segment flag that needs a parser span type (`+c6`); a
manual-vs-binary contradiction escalated to the maintainer
(`+xWORD`/`+x@FILE`, `+a`); and output-routing infra (`+f`). The
consolidated assessment for the maintainers, including the cost estimate
for porting the rest of CLAN, is in the FREQ porting report.

The byte-parity numerator in the table above (25 / 44) counts only the
rows with a CLAN-re-running golden; the denominator dropped from 47 to 44
as three rows moved from Missing to Deferred (excluded from the
denominator as documented non-goals). The remaining Done rows are
rejections, structural (multi-file orchestration), or spreadsheet
semantic-equivalence cases that are proven at their own seam rather than
by a single-file render golden. With FREQ depth-complete, the next
command (a smaller analysis command, e.g. MLU or KIDEVAL) validates that
the shared primitives make the second command cheap.

## All other commands

Every CLAN command outside FREQ is in one of two states:

1. **Implemented but not depth-first verified.** Most analysis,
   transform, and converter commands have a chatter implementation
   that handles the common-case flags. They have not been driven
   through the byte-for-byte parity gate. Treat their output as
   "approximate", close enough for many uses, but not a drop-in
   replacement for legacy CLAN output in a publication-grade
   workflow.
2. **Intentionally unimplemented.** CLAN's MOR / POST / MEGRASP grammar
   commands are deliberately redirected to the batchalign neural
   pipeline rather than reimplemented as legacy CLAN ports. See the
   command pages for those commands for the deliberate-divergence
   rationale.

## What "done" means here

A command is **done** when every CLAN-documented flag for it has been
classified into one of:

- **matches**: chatter implements the CLAN semantic byte-for-byte,
  with a passing golden test comparing `chatter clan <cmd> <flag>
  <fixture>` to legacy `<cmd> <flag> <fixture>`.
- **deferred**: intentionally not implemented yet, with a written
  source-grounded reason on the command's documentation page.
- **diverged**: chatter intentionally produces a different result,
  with a written source-grounded reason on the command's
  documentation page.

Anything left as `**unverified**`, "matches probably", or "needs
checking" disqualifies the command from "done", every claim must
be source-grounded.

## How this page is updated

The **golden-verified** table is generated and CI-guarded: the
`clan_golden` metric test computes it from the fail-closed golden
tests (the numerator) and the per-command audit-table denominators,
and fails the build if the committed block is stale (regenerate with
`UPDATE_PARITY_METRIC=1`). This is the authoritative, automated signal,
and is exactly the CI automation the acceptance snapshot once deferred
as "a later step".

The **acceptance snapshot** (Category A/B/C/D) is the older,
hand-maintained cut from an external probe that records flag
*acceptance* (exit code), not output parity. It is being superseded
command by command as each gains golden coverage; do not read its
"Done" counts as byte-parity claims.

## Faithfulness rules

Two policies govern how chatter implements each CLAN flag:

1. **Faithfulness rule.** Every CLAN flag slot, `+a`, `+bN`, `+c`,
   `+dN`, `+s`, `+t`, etc., implements CLAN's documented semantic
   byte-for-byte. **Chatter extensions** (researcher conveniences
   not in CLAN, new output formats, ergonomic shortcuts) get a
   chatter-specific flag name (`--list-types`,
   `--include-replacements`, etc.), **never** a CLAN slot. Squatting
   a chatter semantic on a CLAN flag is the worst kind of parity bug
; it looks correct in passing but silently produces non-CLAN
   output for the exact users (CLAN-trained researchers) the
   command targets.

2. **No-speculation rule.** Every claim about CLAN behavior in this
   book or in an audit doc carries a file:line citation from CLAN
   source (e.g. `freq.cpp:690-765`). Every claim about chatter
   behavior carries a file:line citation from chatter source.
   "Likely matches" / "appears invented" guesses are banned unless
   explicitly tagged `**unverified**` with the specific file:line
   that would settle the question.

## Canonical CLAN source for citations

When this book or any audit cites a CLAN behavior at a file:line,
the path resolves inside the
[`TalkBank/OSX-CLAN`](https://github.com/TalkBank/OSX-CLAN) repository
on the `dev` branch, under `src/clan/<file>.cpp`.

`OSX-CLAN/dev` is the TalkBank Git layer over the upstream CLAN
maintainer's latest `unix-clan.zip` from `dali.talkbank.org`,
published there far more often than committed to GitHub. Citations
against `master` of
the same repository are typically stale; always ground at `dev` after
the most recent sync. A sync script imports the latest snapshot onto
`dev` as a single dated commit, so
`git log src/clan/<file>.cpp` on `dev` is the de-facto audit history
of upstream changes.
